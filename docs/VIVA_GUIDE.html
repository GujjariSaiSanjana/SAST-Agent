1 What is Cybersecurity?

Simple Definition

Cybersecurity is the practice of protecting computers, networks, programs and data from unauthorized access, damage or attack. Think of it like a bank vault — the stronger the vault (security), the harder for thieves (hackers) to get in.

🔴 Attackers want to…

  • Steal passwords, credit cards, personal data
  • Take control of systems (ransomware)
  • Bring websites/apps down (DoS)
  • Inject malicious code into software
  • Impersonate users (identity theft)

🟢 Defenders do…

  • Find and fix vulnerabilities before attackers do
  • Scan code for security flaws (SAST)
  • Use encryption, authentication
  • Monitor logs and alerts
  • Follow security standards (OWASP, CWE)

Types of Security Testing

SAST
Static Application Security Testing
Analyses source code without running it. Finds vulnerabilities at code level. Like a spell-checker for security. This project uses SAST.
DAST
Dynamic Application Security Testing
Tests a running application by attacking it. Finds runtime vulnerabilities like a real hacker would. Example: OWASP ZAP.
IAST
Interactive Application Security Testing
Hybrid — runs inside the application while it is being tested. Combines SAST + DAST advantages.
SCA
Software Composition Analysis
Scans third-party libraries/dependencies for known vulnerabilities. Example: npm audit.
Pen Testing
Penetration Testing
Ethical hacker manually tries to break into a system to find weaknesses before real attackers do.
Threat Modelling
STRIDE / PASTA framework
Systematically identifying threats to a system before building it. Answers "what can go wrong?"

2 Cybersecurity Jargon Dictionary

Standards & Frameworks

OWASP Top 10
Open Web Application Security Project
The 10 most critical web security risks. Every developer must know these. e.g., A01: Broken Access Control, A03: Injection, A07: Authentication Failures.
CWE
Common Weakness Enumeration
A numbered catalog of software weaknesses. e.g., CWE-89 = SQL Injection, CWE-79 = XSS. Our AI scanner tags each finding with a CWE ID.
CVE
Common Vulnerabilities and Exposures
Unique identifier for publicly known vulnerabilities in specific software versions. e.g., CVE-2021-44228 = Log4Shell. CWE is the weakness TYPE; CVE is the specific INSTANCE.
CVSS
Common Vulnerability Scoring System
A 0–10 score measuring vulnerability severity. 9–10 = Critical, 7–8.9 = High, 4–6.9 = Medium, 0–3.9 = Low. Our system shows risk scores to users.
SANS Top 25
SysAdmin, Audit, Network, Security
25 most dangerous software errors. Our AI prompt explicitly checks for these alongside OWASP Top 10.
Zero Trust
Zero Trust Architecture
"Never trust, always verify." No user or system is trusted by default — even inside the network. Our auth middleware implements this by validating JWT on every request.

Vulnerability Types

SQL Injection (SQLi)
CWE-89 · OWASP A03
Attacker inserts SQL code into input fields. Example: ' OR 1=1-- logs in without password. Our scanner looks for unparameterized queries.
XSS
Cross-Site Scripting · CWE-79
Attacker injects JavaScript into web pages viewed by other users. Steals cookies, redirects users. e.g., <script>alert(document.cookie)</script>
CSRF
Cross-Site Request Forgery · CWE-352
Tricks a logged-in user into performing unwanted actions. e.g., hidden form that transfers money. Prevented by CSRF tokens.
SSRF
Server-Side Request Forgery · CWE-918
Attacker tricks server into making requests to internal systems. Can expose cloud metadata endpoints (AWS IMDSv1 attack).
IDOR / BOLA
Insecure Direct Object Reference
Attacker accesses another user's data by changing an ID. e.g., /api/users/123 → change to /api/users/124. Our routes check ownership on all DB queries.
RCE
Remote Code Execution · CWE-78
Worst vulnerability — attacker can run arbitrary commands on your server. Often via command injection or deserialization bugs.
Path Traversal
Directory Traversal · CWE-22
Attacker accesses files outside intended directory. e.g., ../../etc/passwd. Our ZIP extraction validates all paths to prevent this.
Hardcoded Secret
CWE-798
Passwords, API keys, tokens stored in source code. Our scanner specifically looks for these patterns. Never commit secrets to git!

Auth & Identity Terms

OAuth 2.0
Open Authorization
Lets users grant third-party apps access to their data without sharing passwords. "Login with GitHub" uses OAuth. User logs into GitHub, GitHub tells our app "this user is verified."
JWT
JSON Web Token
Compact, signed token for authentication. Three parts: Header.Payload.Signature. Our backend issues JWTs stored in HttpOnly cookies after GitHub OAuth.
HttpOnly Cookie
Cookie security attribute
Cookie that JavaScript cannot access. Prevents XSS from stealing session tokens. Our JWT is stored this way for security.
Bcrypt
Password hashing algorithm
Slow, salted one-way hash for passwords. The slowness is intentional — makes brute-force attacks impractical. Our project imports bcryptjs.
Rate Limiting
Request throttling
Blocks users who make too many requests. Prevents brute-force attacks and DoS. Our API: 1000 req/min global, 5 scans/hour, 10 AI explains/minute.
CORS
Cross-Origin Resource Sharing
Browser security feature. Server tells browser which origins can make requests. Our backend only allows requests from localhost:5173.

3 Project Overview

What is SAST Security Copilot?

An AI-powered web platform that automatically scans source code repositories or ZIP archives for security vulnerabilities. Unlike traditional scanners that just list rule violations, this system uses Large Language Models (LLMs) to provide:

  • Human-readable explanations of each vulnerability
  • Real-world attack scenarios (Proof of Concept)
  • Business impact assessment
  • Step-by-step remediation guidance
  • Fixed code snippets ready to apply

🎯 Problem Solved

Traditional SAST tools give raw output like "CWE-89 at line 42" — meaningless to junior developers. This tool converts that into plain-English explanations with AI, making security accessible to everyone.

👥 Target Users

  • Developers who want to secure their own code
  • Security teams doing code reviews
  • DevSecOps pipelines
  • Students learning about application security

⚡ Key Features

  • GitHub OAuth login — no passwords needed
  • Scan via GitHub URL or ZIP upload
  • Real-time progress via Server-Sent Events
  • AI fallback chain (3 providers)
  • Risk score dashboard with charts
  • Project management for multiple codebases

6 Tech Stack Explained

⚛️ React 18 + Vite

React is a JavaScript library for building UIs with reusable components. Vite is a blazing-fast build tool. The frontend is a Single Page Application (SPA) — one HTML file, JavaScript handles all navigation without full page reloads.

🐻 Zustand

Lightweight state management library. The authStore holds the logged-in user globally so any component can access it. Alternative to Redux but far simpler.

🔄 TanStack Query

Data fetching + caching library. Handles loading/error states, re-fetching on window focus, and caching API responses automatically. Replaces manual useEffect fetch patterns.

🎨 Tailwind CSS + shadcn/ui

Tailwind is a utility-first CSS framework. shadcn/ui provides unstyled, accessible UI components (Button, Dialog, Tabs, etc.) built on Radix UI primitives.

📊 Recharts + Monaco

Recharts renders charts (pie charts for severity breakdown, bar charts for risk scores). Monaco is the VS Code editor — used to display code snippets with syntax highlighting.

🟢 Node.js + Express

Node.js runs JavaScript on the server. Express is a minimal web framework for building REST APIs. Our backend handles auth, CRUD operations, file uploads, and SSE streaming.

📘 TypeScript

A typed superset of JavaScript. Catches bugs at compile time (before the code runs). Both frontend and backend use TypeScript for safer, more maintainable code.

🗄️ Prisma ORM

Object-Relational Mapper — lets you write database queries in TypeScript instead of raw SQL. Schema defined in schema.prisma. Migrations track schema changes over time.

🐘 PostgreSQL 15

Relational database. Stores users, projects, scans, and issues. Chosen over MySQL for better JSON support, full-text search, and reliability. Runs in Docker.

⚡ Redis 7

In-memory data store. Used as the job queue backend for Bull. Jobs (scan tasks) are stored in Redis so they survive backend restarts. Also supports job retry, timeout, and monitoring.

🐂 Bull Queue

A Node.js job queue library backed by Redis. Scan jobs are resource-intensive (git clone + AI calls). Bull offloads them from the HTTP request lifecycle — the API responds immediately and the work happens in background workers.

🔐 Passport.js

Authentication middleware for Node.js. Uses "strategies" — passport-github2 for OAuth login and passport-jwt for JWT verification on protected routes.

🛡️ Helmet

Sets security-related HTTP headers automatically. Prevents clickjacking (X-Frame-Options), MIME sniffing (X-Content-Type-Options), and XSS (Content-Security-Policy). One line of code, huge security gain.

✅ Zod

TypeScript-first schema validation library. Validates API inputs (e.g., ensures repoUrl is a valid GitHub/GitLab/Bitbucket URL) and throws structured errors. Prevents invalid data reaching the database.

🐳 Docker + Docker Compose

Docker packages the app into containers — isolated environments with all dependencies. Docker Compose orchestrates all 4 containers (postgres, redis, backend, frontend) with one command: docker compose up.

📝 Winston Logger

Structured logging library. Logs timestamped JSON entries for every request, queue job, and AI call. Essential for debugging production issues.

7 Security Features Implemented

Layered Security (Defence in Depth)

The system uses multiple overlapping security layers — if one fails, others still protect. This is the "defence in depth" principle.

🔒 Authentication — GitHub OAuth + JWT

  • No local passwords — reduces credential theft risk
  • JWT stored in HttpOnly cookie (JS can't read it → no XSS theft)
  • SameSite=Lax on cookies → CSRF protection
  • Secure flag in production → HTTPS only
  • 7-day token expiry — limits exposure window

⏱️ Rate Limiting (3 Tiers)

  • Global: 1000 req/min per IP (brute-force prevention)
  • Scans: 5 scans/hour per user (abuse prevention)
  • AI Explain: 10 requests/min per user (cost control)
  • Returns structured JSON errors, not HTML
  • Health endpoint /health exempt from rate limiting

🌐 CORS Policy

  • Only allows origin http://localhost:5173
  • Credentials (cookies) enabled only for that origin
  • Allowed methods explicitly listed
  • Preflight OPTIONS handled automatically

🪖 Helmet Security Headers

  • X-Frame-Options: SAMEORIGIN → prevents clickjacking
  • X-Content-Type-Options: nosniff → prevents MIME confusion
  • Strict-Transport-Security → force HTTPS
  • Content-Security-Policy → prevents script injection
  • Referrer-Policy: no-referrer → privacy

✅ Input Validation (Zod)

  • All API inputs validated with Zod schemas
  • Repo URL must be GitHub/GitLab/Bitbucket only
  • File size limit: 50MB for ZIP uploads
  • Path traversal prevention in ZIP extraction
  • SQL injection impossible via Prisma (parameterised queries)

🔑 Authorisation (Row-Level Security)

  • Every DB query filters by userId
  • Users can only see their own projects/scans/issues
  • IDOR prevented — changing IDs in URLs returns 404
  • Ownership checked before update/delete operations

📁 Path Traversal Prevention

// ZIP extraction validates every path
if (fileName.includes('..') ||
    path.isAbsolute(fileName)) {
  entry.autodrain(); // skip
}

Prevents attackers from submitting ZIPs containing ../../etc/passwd paths.

🗜️ Compression + Request Size Limits

  • Response compression reduces data transfer
  • JSON body limit: 10MB
  • ZIP upload limit: 50MB
  • Prevents memory exhaustion attacks

8 AI Engine Deep Dive

How AI-Powered SAST Works

Traditional SAST uses rule-based pattern matching (regex). This system uses LLMs to understand code semantics — it can find logical vulnerabilities that rules miss. Each file is sent with a carefully crafted prompt instructing the AI to act as a "hardened senior security auditor."

Fallback Chain Architecture

NVIDIA NIM (Qwen3-235B)
→ fails? →
Google Gemini 1.5 Flash
→ fails? →
OpenAI GPT-4o-mini
→ fails? →
Log & Skip

NVIDIA is tried first (cheapest/fastest). If it fails (rate limit, timeout, invalid JSON), Gemini is tried. If Gemini fails, OpenAI is the last resort. If all fail for a file, that file is skipped but the scan continues. This ensures 99%+ scan completion even if one provider is down.

What the AI Prompt Contains

The AI is given the role of a "Hardened Senior Security Auditor" and instructed to check for:

OWASP Top 10 & SANS Top 25
SQLi, NoSQLi, XSS, SSRF, CSRF, IDOR, Broken Auth, Insecure Deserialization
Cryptographic Failures
Weak algorithms, hardcoded secrets, missing salt, bypassable encryption
Business Logic Flaws
Race conditions, unauthorized state transitions, price manipulation
Input Validation
Missing output encoding, improper regex, buffer overflows
Configuration Issues
Leaked credentials in comments, insecure default settings, dangerous APIs
Supply Chain
Malicious patterns, insecure dependency usage

Two-Phase AI Processing

Phase 1: AI Scanning

File-by-file scan. Each file's content is sent to AI, which returns a JSON array of findings. The AI must respond in strict JSON format — invalid responses are parsed and cleaned automatically.

Output per finding: ruleId, title, description, severity, category, lineStart, lineEnd, codeSnippet, cweId, owaspId

Phase 2: AI Enrichment

After all raw findings are saved, a second AI pass enriches each issue with deeper analysis. Batch of 5 issues, 2 concurrent workers to balance speed vs API rate limits.

Adds: explanation, impact, proofOfConcept, remediation, fixCode

Risk Score Calculation

Each completed scan gets a 0–10 risk score based on issue severity counts:

Risk Score = (Critical×10 + High×5 + Medium×2 + Low×0.5) / totalIssues (capped at 10)

Higher score = more critical vulnerabilities found. Displayed as a gauge on the dashboard.

SSE — Server-Sent Events Explained

What is SSE?

A one-way stream from server to client over a single HTTP connection. The frontend subscribes to /api/scans/:id/stream and receives JSON events every second as the scan progresses.

Why SSE over WebSockets?

SSE is simpler — just HTTP. Works through proxies and firewalls. No special server setup. Since we only need server → client updates (not bidirectional), SSE is the right choice.

What is SAST and how does it differ from DAST?

SAST (Static Application Security Testing) analyses source code without executing it. It reads the code and finds vulnerabilities like a linter. Can run early in development (shift-left security).

DAST (Dynamic Application Security Testing) attacks a running application from the outside, like an automated hacker. Finds runtime vulnerabilities SAST misses (e.g., configuration issues, runtime injection).

Key difference: SAST = white-box (sees code), DAST = black-box (sees only behaviour). This project implements SAST — it reads the code files directly.

What is OWASP Top 10? Name at least 5 categories.

OWASP (Open Web Application Security Project) Top 10 is the authoritative list of the most critical web application security risks:

  • A01: Broken Access Control — Users can access data/functions they shouldn't
  • A02: Cryptographic Failures — Weak encryption, data transmitted in plaintext
  • A03: Injection — SQL, command, LDAP injection attacks
  • A04: Insecure Design — Missing security in architecture phase
  • A05: Security Misconfiguration — Default passwords, open cloud storage
  • A06: Vulnerable Components — Outdated libraries with known CVEs
  • A07: Auth Failures — Weak passwords, missing MFA, session fixation
  • A08: Integrity Failures — Unsigned updates, insecure deserialization
  • A09: Logging Failures — No audit trail, attackers can cover tracks
  • A10: SSRF — Server tricked into fetching internal resources

Our AI scanner prompt explicitly checks for all 10 categories.

What is the difference between CWE, CVE, and CVSS?

CWE (Common Weakness Enumeration): A catalog of types of software weaknesses. e.g., CWE-89 = SQL Injection (as a category). Maintained by MITRE.

CVE (Common Vulnerabilities and Exposures): A specific instance of a vulnerability in a specific software version. e.g., CVE-2021-44228 is the Log4Shell bug in Log4j 2.x. Has a CVE ID, description, and affected versions.

CVSS (Common Vulnerability Scoring System): A 0–10 numerical score for a CVE based on exploitability and impact. 9.8 = Critical. Used to prioritise patching.

Analogy: CWE is "robbery as a category." CVE is "the robbery at 123 Main St on 1 Jan 2024." CVSS is "how severe was that robbery (10/10 = armed, no witnesses)."

How does GitHub OAuth work in this project?
  1. User clicks "Login with GitHub" → Frontend hits GET /auth/github
  2. Backend (Passport.js) redirects user to github.com/login/oauth/authorize?client_id=xxx
  3. User approves on GitHub. GitHub redirects back to /auth/github/callback?code=xxx
  4. Backend exchanges the code for an access_token by calling GitHub's API
  5. Backend fetches user profile (username, email, avatar) from GitHub API
  6. Backend upserts user in PostgreSQL (create if new, update if existing)
  7. Backend generates a JWT signed with JWT_SECRET, stored in HttpOnly cookie
  8. User is redirected to /dashboard — now authenticated
  9. Every subsequent API request includes the cookie; Passport-JWT middleware validates it
What is a JWT and why use HttpOnly cookies for storage?

JWT (JSON Web Token) has 3 base64-encoded parts: Header.Payload.Signature

  • Header: Algorithm used (e.g., HS256)
  • Payload: Claims — userId, username, expiry
  • Signature: HMAC of Header+Payload using secret — proves authenticity

Why HttpOnly cookies? The alternative is localStorage, but JavaScript can read localStorage — meaning an XSS attack could steal the token. HttpOnly cookies cannot be read by JavaScript at all. Only the browser sends them automatically with requests. This is the more secure approach.

Combined with SameSite=Lax, the cookie isn't sent on cross-site requests, preventing CSRF attacks.

Why use a Bull Queue and Redis? Why not just process scans directly?

Scans are long-running, resource-intensive tasks — cloning a repo can take 30 seconds, AI analysis of 100 files can take 5+ minutes. If we processed this directly in the HTTP request handler:

  • The HTTP request would time out (typically 30s)
  • The user's browser would show an error
  • Server would be blocked, unable to serve other requests
  • If server restarts, the scan is lost

With Bull Queue + Redis:

  • API responds immediately with {scanId, status: PENDING}
  • Job persists in Redis — survives server restarts
  • Workers process jobs concurrently in background
  • Frontend polls progress via SSE
  • Job retries, timeouts (15min), and monitoring built-in
What is SQL Injection and how does Prisma prevent it?

SQL Injection is when an attacker inserts SQL code into input fields to manipulate database queries.

-- Vulnerable raw query:
SELECT * FROM users WHERE username = '${userInput}'
-- Attacker inputs: ' OR '1'='1
-- Result: SELECT * FROM users WHERE username = '' OR '1'='1'
-- This returns ALL users!

How Prisma prevents it: Prisma uses parameterized queries internally. You write:

prisma.user.findFirst({ where: { username: userInput } })

Prisma converts this to a prepared statement where the user input is never concatenated into SQL. The database treats it as data, not code. SQL injection is impossible through the Prisma ORM.

Explain the multi-stage Docker build and why it matters.

A standard Dockerfile ships everything including devDependencies (TypeScript compiler, ts-node, jest, etc.) — making a huge image (often 1–2 GB).

Our multi-stage backend Dockerfile:

  1. Builder stage (node:20-alpine): Install ALL deps, compile TypeScript → JavaScript, generate Prisma client
  2. Runner stage (node:20-alpine, fresh): Install only production deps, copy dist/ and .prisma/client from builder

Benefits:

  • Smaller image — no TypeScript compiler, no source code, no devDeps
  • Faster startup — less to load
  • More secure — smaller attack surface (fewer packages)
  • Production image doesn't expose source TypeScript
What is rate limiting and why are there 3 different rate limiters?

Rate limiting caps the number of requests a client can make in a time window. Different endpoints have different risk profiles:

LimiterLimitWhy
Global1000 req/min per IPPrevents DoS, brute-force attacks on any endpoint

AI Explain10 req/min per userOn-demand AI calls cost money — prevents abuse

The scan and AI limiters key on user ID (not IP) — so users behind the same office network aren't unfairly limited together.

What is IDOR and how does this project prevent it?

IDOR (Insecure Direct Object Reference): When an application uses user-supplied input to access objects directly without checking permissions.

Example: User A has scan ID abc123. User B tries GET /api/scans/abc123. Without protection, User B sees User A's results.

How we prevent it: Every single Prisma query includes { where: { id, userId } }:

prisma.scan.findFirstOrThrow({
  where: { id: scanId, userId: req.user.id }
})

If the scan exists but belongs to a different user, Prisma throws a "not found" error. The attacker gets a 404, not the data. This is row-level authorisation.

What is the Strategy Pattern and where is it used here?

The Strategy Pattern defines a family of algorithms, encapsulates each one, and makes them interchangeable. The client uses the algorithms without knowing their implementation.

In this project, the AiProvider interface is the strategy:

interface AiProvider {
  name: string;
  scan(filePath: string, content: string): Promise<AiScanFinding[]>;
  analyze(issue: AiIssueInput): Promise<AiAnalysisResult>;
}

NvidiaProvider, GeminiProvider, and OpenAIProvider all implement this interface. The AiScannerService and AiService hold an array of providers and call them without knowing which specific AI company's API they're using. Adding a 4th AI provider requires only implementing the interface — zero changes to the rest of the system.

What is Server-Sent Events (SSE) and why use it for scan progress?

SSE (Server-Sent Events) is a browser standard that allows a server to push data to a client over a persistent HTTP connection. Unlike WebSockets (bidirectional), SSE is unidirectional (server → client only).

The frontend opens a connection to /api/scans/:id/stream:

// Backend sends:
res.setHeader('Content-Type', 'text/event-stream');
res.write(`data: ${JSON.stringify(progressData)}\n\n`);

Why SSE for scan progress?

  • Scans take minutes — polling every second wastes bandwidth with full HTTP overhead
  • SSE keeps one connection open, server pushes updates when they happen
  • Simpler than WebSockets — no special protocol, works through proxies
  • Browser auto-reconnects if connection drops
Why does the ZIP extraction check for path traversal?

Path Traversal (Directory Traversal) is an attack where a malicious ZIP file contains entries with paths like ../../etc/passwd or ../backend/.env. When extracted naively, these write files outside the intended directory — potentially overwriting sensitive system files or application secrets.

// Our protection:
if (fileName.includes('..') || path.isAbsolute(fileName)) {
  logger.warn(`Blocked ZIP path traversal: ${fileName}`);
  entry.autodrain(); // skip this entry
  return;
}

Any ZIP entry containing .. (parent directory traversal) or an absolute path (/etc/passwd) is silently skipped. The rest of the ZIP is still processed safely. This prevents CWE-22.

What is Prisma and why use an ORM instead of raw SQL?

Prisma is a next-generation ORM (Object-Relational Mapper) for Node.js and TypeScript.

Benefits over raw SQL:

  • Type safety: Auto-generated TypeScript types from your schema — IDE catches wrong column names at compile time
  • SQL injection prevention: All queries are parameterised automatically
  • Migrations: Schema changes are tracked as versioned migration files
  • Readable syntax: prisma.user.findMany({ where: { email } }) is clearer than raw SQL
  • Prisma Studio: Visual database browser for development

The schema is defined once in schema.prisma. Running prisma generate creates fully-typed client code. Running prisma migrate deploy applies changes to the database.

What is the purpose of the AI prompt engineering in this system?

Prompt engineering is crafting instructions that get reliable, structured output from an LLM. Our scan prompt uses several techniques:

  • Role assignment: "You are a hardened senior security auditor" — puts the AI in the right mindset
  • Explicit scope: Lists exactly what to check (OWASP, cryptography, SSRF, etc.)
  • Strict output format: "Respond ONLY with a valid JSON object with a findings array" — ensures parseable output
  • Anti-hallucination rules: "Do NOT report generic warnings, only actionable security risks" — reduces noise
  • Schema specification: Each field of the JSON is defined with expected values
  • Zero-trust stance: Assumes all code is potentially malicious until proven safe

The response parser also handles malformed JSON by stripping markdown code fences before parsing.

What is Zod and how is it used for validation?

Zod is a TypeScript-first schema validation library. You define a schema once and it validates data + infers TypeScript types simultaneously.

export const CreateScanSchema = z.object({
  repoUrl: z.string().url().refine(
    (url) => url.includes('github.com') ||
              url.includes('gitlab.com'),
    { message: 'Must be GitHub/GitLab/Bitbucket URL' }
  ),
  projectId: z.string().optional(),
});

The controller calls CreateScanSchema.parse(req.body) — if validation fails, Zod throws a structured error with field-level messages. Our error handler converts this to a clean JSON 400 response. This ensures no invalid data reaches the database or queue.

What is the difference between authentication and authorisation?

Authentication = "Who are you?" — verifying identity. Our system uses GitHub OAuth + JWT. The JWT proves you are who you claim to be.

Authorisation = "What are you allowed to do?" — checking permissions. After authentication, every data access in our system checks if the authenticated user owns the resource:

// Both conditions must match:
prisma.scan.findFirstOrThrow({
  where: {
    id: scanId,      // correct resource
    userId: user.id  // belongs to this user
  }
})

This is why you can log in (authenticated) but can't see someone else's scan (not authorised). The middleware checks authentication; the service layer checks authorisation.

What is the fallback chain and why is it important for reliability?

The fallback chain tries AI providers in priority order. If one fails, the next is used:

NVIDIA NIM
→ fail →
Gemini
→ fail →
OpenAI

Failures can happen due to: rate limits, API downtime, invalid JSON response, network timeout.

Why important? AI APIs are not 100% reliable. Without fallback, a single provider outage would make all scans fail. With the chain, the system degrades gracefully — a file that fails all 3 providers is logged and skipped, but the rest of the scan completes normally. This is the Fault Tolerance principle.

NVIDIA is primary because it's the configured custom model (Qwen3-235B via NVIDIA NIM) which may be cheapest. Gemini and OpenAI are fallbacks with different API quotas, so it's unlikely all three fail simultaneously.
