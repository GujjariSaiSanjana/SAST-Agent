00 Your Opening Pitches

Faculty always opens with "tell me about your project." Have these three versions ready. The 30-second one is non-negotiable — memorise it.

One-Sentence Pitch

"An AI-powered tool that automatically scans software source code for security vulnerabilities and explains each issue to the developer in plain English — before the software is released."

Explain Like I'm 5

Imagine a very smart spell-checker, but instead of checking spelling, it reads code and says "this part is dangerous — a hacker could use it to steal passwords." Then it tells you exactly how to fix it.

The Core Analogy

SAST is like a building inspector who visits before construction is finished. They read the blueprints and spot structural flaws — before the building opens and before anyone gets hurt. Our AI explains the inspection report in simple language.

30-Second Pitch — Memorise This

"Our project is an AI-powered security tool for developers. When you write code and push it to GitHub, there are often security mistakes hidden in it — things hackers can exploit. Our tool automatically scans that code, finds those mistakes, and then uses AI to explain what each problem is, why it's dangerous, and exactly how to fix it. It runs as a web application — you log in with GitHub, submit your repository, and in seconds you get a prioritised list of security issues with plain-English explanations. No security expertise required to use it."

~30 seconds at normal pace
2-Minute Full Explanation

"Software security is one of the most critical and most neglected areas of modern development. Every application has code — and that code can have flaws that attackers exploit to steal data, crash systems, or take over accounts. The IBM Cost of Data Breach Report 2024 puts the average breach at $4.88 million. Most of those breaches exploit vulnerabilities that already existed in the code before it went live.

The standard solution is security testing — but the professional tools are complex, expensive, and produce technical reports that only security specialists can interpret. A developer at a startup or a student building an application has no practical way to know if their code is secure.

Our project solves this with a two-layer approach. The first layer is a SAST engine — Static Application Security Testing — which reads the source code without running it and identifies patterns that match known vulnerability categories defined by OWASP and CWE, the globally recognised security standards bodies. The second layer is AI enrichment: for every vulnerability found, we send the details to a large language model (via NVIDIA NIM, Google Gemini, or OpenAI — in a priority fallback chain) which generates a plain-English explanation, a risk assessment, and a remediation code snippet.

The whole system is a web application. Developers log in via GitHub OAuth, submit a repository link or zip file, and see real-time scan progress in a dashboard. Results are stored permanently so teams can track their security posture over time. Everything runs in Docker containers — meaning it deploys anywhere with a single command."

~2 minutes at normal pace

01 Overview & One-Liners

Faculty always opens with "tell me about your project." Have a 30-second version, a 2-minute version, and a one-liner ready.

snobby So, what exactly is your project? Explain it in one sentence.
Your answer: "Our project is an AI-powered tool that automatically scans software source code, finds security vulnerabilities before the software is released, and explains each vulnerability to the developer in plain English."

If they push for simpler: "Think of it as a smart antivirus — but instead of scanning files for malware after a computer gets infected, ours scans the code itself before the software even runs, and catches the mistakes that hackers would exploit."
snobby What does SAST stand for? And why that name?
SAST stands for Static Application Security Testing. "Static" means we analyse the code without running it — like reading a recipe for errors before cooking, rather than tasting the dish after something goes wrong. It is the industry standard term used by companies like Google, Microsoft, and all major cybersecurity firms.
confused Is this a website? An app? What exactly did you build?
We built a web application — you open it in a browser like any website. You log in, paste a link to your code repository (or upload code), click Scan, and within seconds it shows you a list of security issues found in the code, with AI-generated explanations and suggested fixes for each issue. There is also a real-time progress bar while the scan runs.
snobby What problem does this solve? Is there really a need for this?
Cybersecurity breaches cost businesses $4.88 million on average per incident in 2024 (IBM Cost of Data Breach Report). The majority of breaches exploit vulnerabilities that were already present in the code — meaning they could have been caught before release. Our tool addresses exactly that gap: automated detection at the development stage, when fixing a bug costs 100× less than fixing it after deployment. The need is not hypothetical — companies like Snyk and Veracode are valued at billions of dollars solving this exact problem.
confused Who is the target user of this application?
Our primary users are software developers and development teams — specifically those who write code and want to know if it has security flaws before they ship it to customers. Secondary users are security engineers or team leads who want a project-wide view of all vulnerabilities. The tool is designed to be used during the software development process, not after the product is already live.

02 Motivation & Innovation

snobby What is the novelty of your project? Tools like this already exist, no?
Yes, enterprise tools exist — Snyk, SonarQube, Veracode. But our novelty is threefold:

1. AI explanation layer — existing open-source scanners just flag a line number and a rule ID. Ours uses a large language model to explain why that code is dangerous and how to fix it in plain language, making it accessible to junior developers who may not understand raw security alerts.

2. AI fallback chain — we use three different AI providers (NVIDIA, Google, OpenAI) in sequence, so if one fails the next takes over automatically, ensuring reliability without vendor lock-in.

3. Integrated, accessible platform — enterprise tools cost hundreds of dollars per month. Ours is a self-hosted, open architecture. The integration of scanning + AI explanation + GitHub authentication in a single modern interface is the differentiation.
confused Why did you choose cybersecurity as your domain?
Cybersecurity is one of the fastest growing fields in technology — the global market is projected to reach $657 billion by 2030. Every application written today needs security testing, and there is a significant shortage of security professionals. Building an AI-assisted tool in this domain gives us real-world relevance, aligns with industry trends, and addresses a concrete, measurable problem that companies actively spend money on.
snobby Why not just use ChatGPT to check the code? Why build a whole system?
ChatGPT is a general-purpose chatbot. It has no structured way to scan a full project with hundreds of files, no persistent storage of results, no queue system for handling multiple users simultaneously, no structured output format tied to industry vulnerability standards (OWASP, CWE), and no integration with GitHub repositories. Our system uses the AI only for the explanation and suggestion step — the actual detection is done by a dedicated, rule-based security analysis engine. The two complement each other: the scanner finds the issue precisely, the AI explains it clearly.
trap How is this different from a simple antivirus?
An antivirus scans compiled programs or files for known malicious signatures at runtime — it looks for malware that already exists. Our tool scans source code — the human-written instructions — before the software is even compiled or run. It finds coding mistakes that could be exploited by an attacker, not pre-existing malware. The difference is: antivirus is a cure, SAST is prevention. You cannot run an antivirus on source code — different problem, different solution.

03 Tech Stack Questions

Faculty read your tech stack on the cover page and will randomly pick something and ask why. These are the exact questions they will ask.

confused You used Node.js for the backend. Why not Python? Python is more popular for AI.
This is a great point and a common misconception. We use Python AI services through APIs — NVIDIA's, Google's, OpenAI's — we are not building or training the AI ourselves. Our backend is the web server that handles user requests, manages the database, schedules jobs, and calls those AI APIs. For this task, Node.js is ideal: it is built for handling many simultaneous web requests efficiently (it is what Netflix, LinkedIn, and Uber use for their servers). Python would be slower for the web server role. We get Python-powered AI without writing Python — by calling their APIs.
confused You have two databases — PostgreSQL and Redis. Why two? Is one not enough?
They serve completely different purposes. ≈ analogy
PostgreSQL is a filing cabinet — permanent, organised, searchable storage. Redis is a whiteboard — fast, temporary, gets wiped when you're done. You wouldn't store a legal contract on a whiteboard, and you wouldn't look up today's queue of tasks in a filing cabinet.
PostgreSQL permanently stores everything: user accounts, scan results, vulnerability reports. Redis holds temporary, fast-access data: pending scan jobs, session tokens, rate-limit counters. It is extremely fast but stores nothing permanently. Using the right tool for each job is standard practice — the same approach used by Twitter, GitHub, and most large platforms.
confused What is Prisma? Is it different from PostgreSQL?
PostgreSQL is the actual database — the storage engine. Prisma is an ORM (Object Relational Mapper). ≈ analogy
PostgreSQL is the bank vault. Prisma is the banker — the intermediary who knows the vault's layout and handles all the paperwork. You tell the banker "store this" or "retrieve that", and they handle the vault mechanics. Without the banker, you'd have to memorise every combination and procedure yourself.
Prisma lets our application talk to PostgreSQL using JavaScript objects instead of raw SQL queries. It also auto-generates the database tables from our schema definition, reducing errors and speeding up development.
confused Why GitHub login only? What if someone does not have a GitHub account?
Our target users are software developers — and virtually every developer today has a GitHub account. It is the industry standard platform for storing and managing code, with over 100 million registered users. By using GitHub OAuth, we also get an immediate benefit: once the user logs in, we can directly access their GitHub repositories with their permission, making the scan workflow seamless. Adding traditional email/password login is a straightforward extension for a future version, but for our primary use case, GitHub login is the most relevant choice.
confused What is Docker? Why did you need it? Can't you just run the application normally?
You can — but it would only work on the one computer where you set it up. ≈ analogy
Docker is like a shipping container. Before containers existed, every cargo ship had to be specially loaded for each port. A shipping container means you pack it once — and every port handles it the same way, regardless of ship or country. Docker does this for software: pack once, run anywhere.
Docker packages the entire application — code, settings, libraries, database — into a self-contained unit. It eliminates the "it works on my machine" problem. Our project has three components (backend, database, job queue) that need to work together — Docker Compose starts all of them with a single command: docker compose up. This is industry-standard practice; virtually every cloud deployment uses containers.
confused What is TypeScript? Why not just use JavaScript?
JavaScript is the standard language for web development but it has no type safety — it will happily let you write code that will crash at runtime. TypeScript adds a type system on top: it catches those errors before the code even runs, during development. For a security application where data integrity matters — user credentials, vulnerability reports, scan results — TypeScript's strictness is not just convenient, it prevents entire categories of bugs. It is the industry standard for large-scale JavaScript projects: used by Google, Microsoft (which created it), Slack, and Airbnb.
trap What is an API? Explain it simply.
An API (Application Programming Interface) is a defined way for two software programs to communicate. Think of a restaurant: the menu is the API — it defines what you can order, in what format, and what you will receive in return. You (the frontend) place an order (send a request) through the waiter (the API), and the kitchen (the backend or AI service) prepares exactly what you asked for and sends it back. In our project, the frontend calls our backend API to submit scans, retrieve results, and request AI explanations. Our backend in turn calls NVIDIA's/Google's API to get AI-generated text.
confused What is React? Why not plain HTML?
Plain HTML is a static document — it cannot update itself without reloading the page. React is a JavaScript framework that creates dynamic interfaces: when a scan result arrives, only the result panel updates, without the entire page refreshing. Our application has real-time progress updates, live-updating vulnerability lists, and code editors embedded in the page — none of these are possible with plain HTML. React is used by Facebook (who created it), Instagram, WhatsApp, Netflix, and virtually every modern web application.

04 UML & Diagram Questions

The classic "why did you put that in the diagram" — faculty point at something random and ask why it's there. Prepare for each component.

Real example from another student's viva:

Student mentioned FastAPI in tech stack and included it in UML. Faculty asked "why did you mention that in the diagram?" The student panicked. Correct response: "Because it is an active component in the system that other components communicate with — in UML, every component that sends or receives data belongs in an architecture diagram." Stay calm, the question is about your confidence, not your knowledge.

snobby Why did you include GitHub in your architecture diagram?
GitHub is an active external component in our system — it is not just mentioned for context. When a user submits a repository link for scanning, our backend clones that repository directly from GitHub over the network. Our authentication system also communicates with GitHub's OAuth servers to verify a user's identity. Any system component that sends or receives data must appear in the architecture diagram — omitting GitHub would make the diagram incomplete and misleading.
confused Why is Redis in the diagram? I thought it was just a database.
Redis appears in the diagram because it plays an active, distinct architectural role — it is the message queue system that holds pending scan jobs. When a user submits a scan, the backend does not immediately run it (that would freeze the server for the next user). Instead it places the job in a Redis queue. A separate background worker picks jobs from that queue and processes them one by one. This is called a queue architecture, and Redis is the component that enables it. Every component with a distinct function and communication path is shown in an architecture diagram.
snobby Can you explain your use case diagram?
Our use case diagram shows two actors: the Developer (primary user) and the System/AI (automated actor). The developer can: register/login, submit a repository for scanning, view scan results, view individual vulnerabilities, and request AI explanation. The System automatically: clones the repository, runs the static analysis engine, stores results in the database, and when requested, calls the AI API to generate explanations. The diagram captures every meaningful interaction between a human user and our application.
confused Why did you make so many diagrams? Was that necessary?
Each diagram communicates a different dimension of the system. The architecture diagram shows how components connect. The sequence diagram shows the time-ordered flow of a specific action. The class diagram shows data structure and relationships. The state diagram shows how the system changes over time. In software engineering, UML (Unified Modelling Language) is the professional standard for documenting systems — different stakeholders (developers, managers, testers) read different diagrams. Having all of them demonstrates that the system was properly designed, not just coded.
snobby What is a sequence diagram? Explain yours.
A sequence diagram shows the step-by-step message flow between components for a specific scenario, in time order (top to bottom). Our scan submission sequence shows: (1) Developer sends POST /scans to the Backend. (2) Backend validates the request and saves a pending scan record to PostgreSQL. (3) Backend pushes a job to the Redis Bull Queue. (4) The Queue Worker picks up the job, clones the GitHub repository, runs the security analysis tool, and saves results to PostgreSQL. (5) Frontend receives real-time progress updates via Server-Sent Events. (6) Final results are displayed to the user. Each arrow represents one API call or function call.

05 AI & Machine Learning Questions

Faculty hear "AI" and immediately ask about training data, accuracy, and "what if it's wrong." They often conflate all AI as one thing.

confused How did you train your AI model? What data did you use?
We did not train an AI model — and that is a deliberate design decision, not a limitation. Training a large language model requires millions of dollars and months of compute time on thousands of GPUs. Instead, we use three production AI services — NVIDIA NIM (running Qwen3-235B), Google Gemini 1.5 Flash, and OpenAI GPT-4o-mini — via their APIs. These are already trained, state-of-the-art models that have seen enormous amounts of code and security documentation in their training. Our value-add is in how we integrate these models: we craft precise security-context prompts, handle failures gracefully with fallback logic, and embed the output in a structured user workflow.
trap What if the AI gives a wrong explanation or a wrong fix? Is it reliable?
This is an important and valid concern. Our architecture accounts for it in two ways:

First, the vulnerability detection is not done by AI. It is done by a deterministic, rule-based static analysis engine. The AI only explains and suggests fixes — it does not decide what is a vulnerability. So even if the AI explanation is slightly imprecise, the underlying finding is reliable.

Second, AI-generated suggestions in our system are presented as guidance, not automated changes. The developer reads the explanation and fix suggestion, then makes their own judgement before modifying code. This is exactly how tools like GitHub Copilot and SonarQube's AI features work — human in the loop, AI as assistant.
confused You mentioned three AI providers. Why three? Is one not enough?
Reliability. Any single API can go down due to server issues, rate limits, or outages. Our fallback chain means: we first try NVIDIA NIM (highest capability). If that fails or times out, we automatically try Google Gemini. If that also fails, we fall back to OpenAI GPT-4o-mini. The user sees a smooth response regardless of which provider actually answered. This is called a resilient, vendor-agnostic architecture — the same principle airlines use with redundant systems: the passenger never sees which engine is working, they just arrive safely.
snobby What is the accuracy of your AI?
Accuracy of explanation quality is subjective — the AI is producing natural language, not binary classifications, so traditional accuracy metrics do not directly apply. What we can measure: our vulnerability detection accuracy (the scanner component, not AI) follows the rule-base accuracy of the underlying engine, which is tested against known vulnerability datasets. The AI's code suggestion quality matches production results from the same models evaluated by Google and OpenAI on coding benchmarks: GPT-4o-mini scores 87% on HumanEval, Gemini 1.5 Flash scores 71%. These are the same models our users receive.
confused What is NVIDIA NIM? Did NVIDIA give you permission to use this?
NVIDIA NIM (NVIDIA Inference Microservices) is a publicly available, commercial AI API service — anyone can sign up and use it, similar to how anyone can use Google Maps API or Twilio for SMS. We registered for an API key through NVIDIA's developer portal, just as developers do worldwide. No special permission is required beyond a standard API account. It provides access to powerful AI models optimised to run on NVIDIA's GPU infrastructure — we send a text request and receive a text response, we do not manage any hardware.

06 Security & Trust Questions

snobby Is your application itself secure? How can a security tool be trusted if it is not secure?
Valid point, and we addressed it directly. Our application implements:

• JWT tokens in HttpOnly cookies — login sessions are stored in a way that JavaScript on the page cannot access, preventing token theft via XSS attacks.
• Rate limiting — limits requests per IP to prevent abuse and brute-force attacks.
• Input validation — all user inputs are validated and sanitised before being processed.
• GitHub OAuth — we never store passwords; authentication is delegated to GitHub's secure infrastructure.
• Parameterised queries via Prisma ORM — prevents SQL injection attacks.

These cover the OWASP Top 10 — the globally recognised list of the most critical web application security risks.
confused What is OWASP? You mentioned it in your report.
OWASP stands for Open Worldwide Application Security Project. It is a non-profit foundation that publishes the most widely referenced security standards in the industry. Their most famous document is the OWASP Top 10 — a list of the 10 most critical and common web application vulnerabilities, updated every few years based on real-world breach data. When our scanner detects a vulnerability, it labels it with the relevant OWASP category (e.g., "A01:2021 Broken Access Control") so developers immediately understand the severity and context within industry-standard classification.
confused What is JWT? Why not just use sessions?
JWT (JSON Web Token) is a compact, self-contained token that encodes user information and is cryptographically signed. Traditional server sessions require the server to store session data in memory or a database for every logged-in user. JWT is stateless — the token itself contains the information, and the server just verifies the signature. This scales better: any server instance can verify any token without needing to query a shared session store. For a web application where the frontend and backend may be on different servers, JWTs are the standard modern approach.
trap If someone submits a malicious repository, could it harm your server?
We considered this threat. The mitigations: (1) The code is cloned into an isolated temporary directory with restricted permissions. (2) The scanner reads files — it does not execute any code from the scanned repository. (3) The scanning process runs in a separate worker process, not the main web server. (4) Temporary files are deleted after scanning. For production hardening, the scan worker would run inside a sandboxed Docker container with no network access — a standard defence-in-depth approach. This is a known challenge in the SAST industry and our architecture follows the established pattern.

07 Business & Real-World Questions

snobby Can this be commercially deployed? Is it production-ready?
As a fully functioning prototype, yes — every feature works end-to-end. For commercial production, additional work would be needed: a billing system, a more aggressive sandboxing layer for the scan worker, enterprise authentication (SAML/SSO), SLA guarantees, and compliance certifications (SOC 2). However, the core architecture — PostgreSQL, Redis queues, stateless JWT auth, Docker deployment — is the same stack used by production SaaS companies. The path from prototype to commercial product is clear and well-understood.
confused What is the cost of running this application?
Infrastructure cost depends on scale. For a small deployment:

• Server hosting (a modest cloud VM): ~$20–40/month
• AI API calls: Pay-per-use. Google Gemini 1.5 Flash costs approximately $0.075 per million tokens — explaining 1,000 vulnerabilities would cost under $1.
• Database: Included in the server, or ~$15/month as a managed service

Total for a small team: under $60/month. This is dramatically cheaper than enterprise SAST tools like Veracode which cost $15,000+ per year. The cost model is a deliberate competitive advantage.
confused How many users can use this simultaneously?
The web server (Node.js) can handle thousands of simultaneous connections with no issue — it is designed for concurrency. The limiting factor is the scan queue: our Bull/Redis queue processes scan jobs sequentially by default (configurable to parallel workers). With a single worker, ten users submitting scans simultaneously would each see their scan complete in order, with a progress indicator while they wait. For higher throughput, you simply add more worker processes — the queue architecture scales horizontally without changing any code.
snobby Why not make a mobile app instead of a website?
Our users are developers working on desktop computers with code editors. They do not write or review code on mobile phones. A web application is available on every device with a browser — mobile, tablet, desktop — without an app store installation. Additionally, the core workflow involves viewing code with syntax highlighting, reading detailed vulnerability reports, and comparing code snippets side by side — these require a large screen and are fundamentally desktop workflows. Building a mobile app would be designing for the wrong user context.
confused What languages can your scanner detect vulnerabilities in?
The underlying scanner engine supports multiple languages including JavaScript, TypeScript, Python, Java, Go, Ruby, and more. The current implementation is focused on JavaScript and TypeScript projects since those are the most common in modern web development, but the architecture is extensible — adding support for additional languages requires adding new rule configurations, not rewriting the system.
snobby What is the contribution of this project to society?
Software security failures have direct societal consequences — hospital systems getting ransomwared and delaying surgeries, personal data of millions being stolen, critical infrastructure being compromised. Most of these start with vulnerabilities in code that were preventable. Our tool makes security scanning accessible to small development teams that cannot afford enterprise tools, democratising a capability that previously required significant resources. Better-secured software across the industry means fewer breaches, less financial harm to individuals, and safer digital infrastructure overall.

08 Testing & Limitations

snobby How did you test this project?
Testing was conducted on multiple levels:

• Unit testing — individual functions (authentication, validation, rate limiting) tested in isolation.
• Integration testing — API endpoints tested end-to-end with a real database (not mocks).
• Functional testing — scanning real open-source repositories with known vulnerabilities to verify detection.
• Vulnerability testing — manually attempting OWASP Top 10 attacks against our own application (SQL injection, XSS, CSRF) to verify our defences.
• Performance testing — submitting multiple concurrent scans to verify the queue system handles load correctly.
trap What are the limitations of your project?
Every well-engineered project has acknowledged limitations — transparency here demonstrates maturity:

• False positives: The scanner may flag code that is technically correct in its specific context — a known limitation of all rule-based static analysis tools.
• No runtime analysis: We only scan code at rest. Some vulnerabilities only appear when code runs (e.g., specific runtime configurations).
• Language coverage: Currently optimised for JavaScript/TypeScript.
• AI cost at scale: API calls have a per-use cost; at very high volume, a self-hosted AI model would be more economical.
• Sandboxing: The scan worker is not fully containerised in the current prototype — a production deployment would add this.
confused Did you scan your own project with your tool?
Yes — and this is actually a standard practice called "dogfooding" (using your own product). We ran our backend codebase through the scanner. The findings were reviewed and addressed, which improved our own code quality. This also served as a real-world validation that the end-to-end workflow — from repository submission to AI-generated fix suggestions — works correctly with a real, non-trivial codebase.

09 Future Scope

Faculty always ask this. Have a crisp list and explain why each is realistic, not just a wish list.

CI/CD Integration

Plug our scanner into GitHub Actions or Jenkins so it automatically scans every code commit and blocks deployment if critical vulnerabilities are found. This is how enterprise SAST tools are used in production pipelines.

More Language Support

Extend scanning to Python, Java, and Go codebases. The queue and AI architecture already supports this — only new rule configurations need to be added per language.

Self-hosted AI Model

At scale, API costs grow. Replacing cloud AI calls with a locally hosted open-source model (e.g., CodeLlama) would eliminate per-call costs and improve data privacy for enterprises.

Vulnerability Trend Dashboard

Track which vulnerability types appear repeatedly across scans over time, helping teams identify systematic coding habits that need team-wide training rather than individual fixes.

Auto-PR Fix Generation

When AI generates a fix, automatically open a pull request in the user's GitHub repository with the corrected code — reducing the manual step of applying the suggestion.

DAST Integration

Add Dynamic Application Security Testing — scanning the running application, not just the code. Combined with SAST, this gives a complete security picture covering both code-level and runtime vulnerabilities.

confused How long did this project take?
State your actual timeline confidently. A reasonable framing: "Active development over [X weeks/months]. The initial architecture and core scanning pipeline took [X weeks]. Integration of the AI explanation layer and the real-time progress system required additional iterations. The final phase covered security hardening, Docker deployment, and documentation." Showing that the project had distinct phases demonstrates structured project management, not just coding.
snobby Did you face any challenges? How did you solve them?
Have 2–3 real challenges ready. Good examples from this project:

1. Infinite authentication loop — the frontend kept reloading because a 401 error triggered a page redirect which triggered another 401. Fixed by removing the redirect from the error interceptor and letting the auth state manager handle it gracefully.

2. AI provider reliability — NVIDIA's API occasionally times out under load. Solution: implemented the three-provider fallback chain with automatic retry.

3. Real-time progress updates — standard web requests cannot push data from server to browser. Implemented Server-Sent Events, which maintains a persistent connection for live scan progress without complex WebSocket overhead.
confused What did you learn from this project?
Strong answer structure: mention a technical learning, a design learning, and a process learning. Example: "Technically, I learned how to design async job queue systems and how to integrate multiple external AI APIs with fallback logic. From a design perspective, I understood why security has to be considered from the start rather than added later — we implemented rate limiting, input validation, and JWT security before writing the business logic. Process-wise, Docker Compose taught me how to think about deployment from day one rather than treating it as an afterthought."

11 Viva Survival Tips

The "I don't know" rule:

If you genuinely do not know, say: "I do not have the exact number off the top of my head, but I can tell you the reasoning behind our approach." Then explain the reasoning. Never guess a statistic — faculty sometimes know the answer and are testing if you will bluff.

On "Why X instead of Y?"

Always answer in terms of fitness for purpose: "X is better suited for [specific requirement] because [concrete reason]. Y would be chosen if [different scenario]." Never say "because we knew X" or "our guide recommended X."

On diagram questions

Faculty point at diagrams randomly. For any component, know: (1) what it does, (2) what it connects to, (3) why it is separate from other components. Say it in 2 sentences. Do not over-explain.

On "is this original?"

Be honest about what you built vs. what you integrated. "We integrated existing AI services and scanner engines, and built the orchestration layer, the user interface, the queue architecture, and the AI fallback logic from scratch." That is real, substantial work.

On "what is the limitation?"

Never be defensive. Name real limitations confidently — it shows engineering maturity. Then immediately follow with: "which we would address in [next version] by [specific approach]."

On snobby tone

Do not match their energy. Stay measured and confident. Short pauses before answering signal composure, not ignorance. Faculty who are snobby are often testing whether you will panic — the student who stays calm and precise usually impresses more than the one who rushes anxious answers.

Opening statement (memorise this)

"Our project is an AI-assisted static application security testing platform. It scans source code repositories, identifies security vulnerabilities using industry-standard rules, and uses large language models to explain each vulnerability and suggest a fix in plain English. The innovation is the AI explanation layer combined with a resilient multi-provider fallback architecture."

Things to absolutely never say:

• "We used this because it was easy to learn." (Say: "because it was the right tool for this requirement.")
• "We didn't have time to implement X." (Say: "X is scoped as a future enhancement.")
• "I'm not sure, my teammate handled that." (Know your entire project.)
• "The AI does everything." (It does not — the scanner does detection, AI only explains.)
• "We trained the AI on our data." (You did not — you called an API.)

The two sentences that save most vivas:

"The decision was driven by [specific requirement]. The trade-off we accepted was [limitation], which we would address by [future approach]." Every "why did you do X?" question is answered by this template.
